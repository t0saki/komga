# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project layout

Komga is a media server for comics/mangas/eBooks. The Gradle root composes three subprojects (only the first two are in `settings.gradle`; `komga-webui` is invoked from the backend via Exec tasks):

- `komga/` — Spring Boot backend (Kotlin, JDK 21 to build, JVM target 17). Hosts the REST API, OPDS v1/v2, Kobo/KOReader sync, and serves the built frontend as static assets.
- `komga-webui/` — Vue 2 + Vuetify 2 + TypeScript SPA. Built at compile time and bundled into the backend jar.
- `komga-tray/` — Jetpack Compose Desktop wrapper that embeds the backend and shows a tray icon (packaged via Hydraulic Conveyor).

Requirements: JDK 21+, Node 18+ (see `.nvmrc`).

## Common commands

Backend (run from repo root):

- `./gradlew bootRun --args='--spring.profiles.active=dev,noclaim'` — run with blank in-memory DB; auto-creates `admin@example.org` / `admin` and `user@example.org` / `user`.
- `./gradlew bootRun --args='--spring.profiles.active=dev,localdb,noclaim'` — same, but persists to `./localdb`.
- `./gradlew :komga:test` — run backend unit/integration tests (JUnit5, profile `test` is auto-applied).
- `./gradlew :komga:test --tests "org.gotson.komga.Some.testName"` — run a single test.
- `./gradlew :komga:benchmark` — run JMH benchmarks (separate `benchmark` source set under `komga/src/benchmark`).
- `./gradlew ktlintFormat` / `ktlintCheck` — Kotlin lint (ktlint 1.7.1). `**/db/migration/**` and generated sources are excluded.
- `./gradlew :komga:jooq-codegen-primary` is the conceptual goal — in practice run `./gradlew :komga:generateJooq` and `:komga:generateTasksJooq` (auto-triggered by `compileKotlin`).
- `./gradlew :komga:prepareThymeLeaf` — build the webui, copy into `komga/src/main/resources/public/`, and rewrite `index.html` with Thymeleaf `th:` tags. Required if you want `bootRun` to serve the latest frontend.
- `./gradlew :komga:bootJar` — produce the runnable jar at `komga/build/libs/komga-<version>.jar`.

Frontend (run from `komga-webui/`):

- `npm install` — install deps (also runnable as `./gradlew :komga:npmInstall`).
- `npm run serve` — dev server on `http://localhost:8081`, proxying API calls to `http://localhost:25600` (the backend). Backend must be started with the `dev` profile or CORS will reject requests.
- `npm run build` — production build into `komga-webui/dist/`.
- `npm run test:unit` — Jest unit tests. Single test: `npm run test:unit -- -t "test name"` or `... path/to/file.spec.ts`.
- `npm run lint` — ESLint (Vue + TypeScript standard configs).

Docker image: `./gradlew prepareThymeLeaf jreleaserPackage` → `komga/build/jreleaser/package/docker/`.

## Spring profiles (composable)

- `dev` — verbose logging, periodic scanning **disabled**, in-memory database, CORS allowed from `localhost:8081`.
- `localdb` — persist database to `./localdb` (override for `dev`).
- `noclaim` — auto-create initial users at startup if none exist; under `dev` this is the fixed admin/user pair, otherwise a random admin password is logged.
- `claim` — used by the OpenAPI generator (`./gradlew :komga:generateOpenApiDocs`) together with `generate-openapi`.
- `test` — applied automatically to JUnit runs (see `withType<Test>` in `komga/build.gradle.kts`).

## Backend architecture

Source root: `komga/src/main/kotlin/org/gotson/komga/`. The package layout is a DDD-style four-layer split — keep new code in the matching layer:

- `domain/` — pure domain `model/`, `persistence/` (interfaces only, no jOOQ), and `service/` business logic. Knows nothing about Spring web or jOOQ.
- `application/` — `tasks/`, `scheduler/`, `events/`. Use-case orchestration above the domain.
- `infrastructure/` — Spring/Jackson/jOOQ/SQLite/Lucene/security/image-IO/etc. wiring. `infrastructure/jooq/` implements the `domain/persistence/` interfaces against generated jOOQ DSL.
- `interfaces/` — entry points: `api/rest`, `api/opds`, `api/kobo`, `api/kosync` (KOReader), `mvc`, `sse`, `apprunner`, `scheduler`. DTOs live under `interfaces/api/dto`.

Entry point: `Application.kt` (`org.gotson.komga.Application`). The tray app's entry point is `org.gotson.komga.DesktopApplicationKt` (in `komga-tray`), which embeds the backend.

### Persistence: two SQLite databases + Flyway + jOOQ

There are two separate SQLite stores with independent Flyway histories and jOOQ packages:

| Logical name | Migrations | jOOQ package | Purpose |
| --- | --- | --- | --- |
| `main` | `komga/src/flyway/{resources,kotlin}/db/migration/sqlite` | `org.gotson.komga.jooq.main` | Library/series/book/user data |
| `tasks` | `komga/src/flyway/resources/tasks/migration/sqlite` | `org.gotson.komga.jooq.tasks` | Background task queue |

jOOQ code is generated by running Flyway against a build-dir SQLite (`build/generated/flyway/{main,tasks}/`) and then `generateJooq` / `generateTasksJooq`. `compileKotlin` depends on `generateTasksJooq`; `generateJooq` should be re-run whenever you add a migration to the `main` schema. Generated sources land in `komga/build/generated-src/jooq/{main,tasks}/` and are excluded from ktlint.

When adding migrations:

- SQL migrations: `db/migration/sqlite/V<N>__<name>.sql` (or the `tasks/migration/sqlite` equivalent).
- Kotlin migrations: live under `src/flyway/kotlin/db/migration/sqlite/` and are compiled in the `flyway` source set before being added to `main`'s outputs.
- `flywayMigrateMain` runs with placeholders (`library-file-hashing=true`, `library-scan-startup=false`, etc.) — placeholder values matter for migrations that reference them.

### Other notable infrastructure

- Search index uses Lucene (`infrastructure/search`).
- Media parsing: Tika + commons-compress + junrar (RAR, including solid rar4) + nightcompress + PDFBox + thumbnailator + twelvemonkeys/nightmonkeys ImageIO plugins (JPEG/TIFF/WebP/JXL/HEIF/JBIG2/JPEG2000) + ZXing for barcodes.
- IDs: `tsid-creator` (TSIDs, not UUIDs).
- Auth: Spring Security with OAuth2 client + form login + Spring Session (Caffeine-backed). Kobo and KOReader sync endpoints have their own auth flows under `interfaces/api/kobo` and `interfaces/api/kosync`.
- OpenAPI: `springdoc-openapi` exposes `/v3/api-docs` and Swagger UI; `./gradlew :komga:generateOpenApiDocs` writes `komga/docs/`.

## Frontend architecture

`komga-webui/src/`:

- `views/` — page-level components routed by `router.ts`.
- `components/` — reusable UI built on Vuetify 2.
- `services/` — Axios-based API clients targeting the backend REST API.
- `store.ts` — Vuex store; persisted via `vuex-persistedstate`, synced with the router via `vuex-router-sync`.
- `i18n.ts` + `locales/` — vue-i18n; translations are managed via Weblate, **do not** hand-edit non-English locale files.
- `plugins/` — Vuetify theme/registration, axios setup, etc.

`vue.config.js` rewrites the dev server's API base URL to `http://localhost:25600`, which is why the backend must run with `dev` profile (for CORS).

## Conventions

- **Commit messages** must follow Conventional Commits (`conventionalcommit.json` enumerates the allowed types). Common scopes are `komga`, `webui`, `tray`. Examples from history: `feat(komga): ...`, `deps(webui): bump ...`, `fix(komga): ...`. `chore` is excluded from changelogs; `perf`, `i18n`, and `deps` get their own changelog sections.
- **Formatting**: `.editorconfig` is authoritative. Kotlin/JS/TS/YAML use 2-space indent; default elsewhere is 4-space. `*.kt` has no max line length.
- Releases are cut by JReleaser (`build.gradle.kts` `jreleaser { ... }`); `CHANGELOG.md` is appended automatically — do not hand-edit.
- `./gradlew test` (or `:komga:test`) before committing, per `DEVELOPING.md`.
